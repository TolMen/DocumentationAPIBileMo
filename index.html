Présentation de l'API BileMo

L'API BileMo est un service conçu pour les entreprises dans le cadre de transactions B2B. Son but principal est de permettre aux plateformes d'accéder facilement au vaste catalogue de produits BileMo, tout en gérant les utilisateurs associés à ces plateformes. Grâce à cette API, les entreprises peuvent intégrer rapidement les produits de BileMo dans leurs propres systèmes et gérer efficacement les utilisateurs qui interagissent avec ces produits.

Public cible

Cette API est spécifiquement conçue pour les entreprises opérant dans un cadre B2B. Elle facilite l'intégration des services BileMo dans les plateformes, en garantissant un accès direct aux produits et une gestion des utilisateurs intuitive.

URL de base

L'API est accessible via l'URL suivante : https://127.0.0.1:8000/
Les entreprises peuvent interagir avec cette API en utilisant cette URL comme point d'accès principal, elle utilise le format JSON.

Fonctionnalités

L'API BileMo offre plusieurs fonctionnalités essentielles qui permettent aux entreprises de gérer leurs interactions avec les produits et les utilisateurs. Les utilisateurs peuvent consulter la liste complète des produits et accéder à leurs détails spécifiques, tout comme ils peuvent visualiser et gérer la liste des clients, ainsi que les utilisateurs qui leur sont liés. De plus, l'API permet d'ajouter, de modifier, ou de supprimer des utilisateurs de manière fluide, offrant ainsi une grande flexibilité dans la gestion des ressources humaines au sein de l'entreprise.

Prérequis techniques

Symfony est le framework PHP utilisé pour construire l'API, offrant une structure robuste pour les API RESTful.

Doctrine ORM facilite la gestion des interactions avec la base de données, en permettant de travailler avec des entités PHP plutôt qu'avec des requêtes SQL directes.

L'API BileMo utilise le mécanisme de JSON Web Tokens (JWT) pour gérer l'authentification et garantir que chaque requête est sécurisée.

Des bases de données comme MySQL ou PostgreSQL sont utilisées pour stocker les produits et les utilisateurs.

Pour tester et interagir avec l'API, Postman est utilisé. Cet outil permet de simuler des requêtes (GET, POST, PUT, DELETE) et d'inspecter les réponses.

Adminer est utilisé pour administrer la base de données.

Authentification avec JWT

L'API BileMo utilise les JSON Web Tokens (JWT) pour sécuriser l'accès aux ressources.
Après authentification réussie, un token JWT est renvoyé, qui doit être inclus dans chaque requête pour accéder aux routes protégées de l'API.

Le token JWT a une durée de vie limitée pour des raisons de sécurité !
Il sera nécessaire de se réauthentifier ou de rafraîchir le token une fois qu'il aura expiré !

Pour obtenir un token JWT, envoyez une requête POST à l'endpoint /api/login_check avec vos identifiants : 

POST /api/login_check
Host : URL de base
Content-Type: application/json

{
    "username": "utilisateur@example.com",
    "password": "mot_de_passe_sécurisé"
}

Si les identifiants sont corrects, vous recevrez un token JWT :

Exemple de réponse 200 OK :

    {
        "token": "eyJhbGciOiJIUzI1NiJ9.eyJ1c2VybmFtZSI6InV0aWx..."
    }

Une fois le token obtenu, incluez-le dans chaque requête vers l'API via le header Authorization avec le préfixe Bearer. 

GET /api/clients
Host : URL de base
Authorization: Bearer <votre_token_JWT>

Si le token est absent, expiré ou invalide, vous recevrez une réponse avec un code d'erreur. Voici un exemple d'erreur lorsque le token est manquant :

Exemple de réponse 401 Unauthorized :
{
    "code": 401,
    "message": "JWT Token not found"
}

Endpoints (URI) de l'API

Voici les différents endpoints disponibles dans l'API
Chaque ressource possède des méthodes spécifiques pour interagir avec les données
Toutes les données des ressources sont mises en cache et expirent au bout de 2 minutes

Ressource : Produits

Pagination disponible par défaut à 10 produits par page
Modifiable avec les paramètres : URI + ?page=X&limit=X (X étant le chiffre voulu)

Méthode Endpoint URI Authentification Description
GET /api/products Token JWT non requis Récupérer la liste des produits
GET /api/products/{id} Token JWT non requis Récupérer un produit par son identifiant

Ressource : Clients

Méthode Endpoint URI Authentification Description
GET /api/clients Token JWT requis Récupérer la liste des clients
GET /api/clients/{id} Token JWT requis Récupérer un client par son identifiant

Ressource : Utilisateurs

Pagination disponible par défaut à 10 utilisateurs par page
Modifiable avec les paramètres : URI + ?page=X&limit=X (X étant le chiffre voulu)

Méthode Endpoint URI Authentification Description
GET /api/users Token JWT requis Récupérer la liste des utilisateurs
GET /api/users/{id} Token JWT requis Récupérer un utilisateur par son identifiant
POST /api/users Token JWT requis Créer un utilisateur lié à un client. 

{
    "email": "nouvel.utilisateur@example.com",
    "password": "motdepasse123",
    "clientId": X
}
PUT /api/users/{id} Token JWT requis Mettre à jour un utilisateur par son identifiant. 

{
    "email": "utilisateur.MODIFIER@example.com",
    "password": "motdepasseMODIFIER"
}
DELETE /api/users/{id} Token JWT requis Supprimer un utilisateur par son identifiant

Gestion des codes HTTP

L'API utilise des codes HTTP standard pour indiquer le succès ou l'échec des requêtes, conformément aux bonnes pratiques REST. Chaque code est associé à une signification spécifique pour aider les développeurs à comprendre et résoudre les problèmes rencontrés.

Signification

La requête a été traitée avec succès.

Réponse

Lorsque la liste des produits ou le détail d'un utilisateur est récupéré correctement.

Signification

Une nouvelle ressource a été créée avec succès.

Réponse

Un body avec les informations du nouvel utilisateur lié à un client ajouté avec succès.

Signification

La requête est mal formée ou contient des données invalides.

Réponse

Une requête POST avec un format de données JSON incorrect ou un paramètre manquant.

Signification

La requête a été traitée avec succès, et il n'y a pas de contenu à renvoyer.

Réponse

Suppression d'un utilisateur, ou quand une ressource a été modifiée et aucune réponse JSON n'est nécessaire.

Signification

Authentification manquante ou token JWT invalide.

Réponse

Le token JWT n’est pas fourni ou il est expiré/invalide.

Signification

Accès refusé, l'utilisateur authentifié n'a pas les droits nécessaires pour accéder à la ressource.

Réponse

Tentative d'accès à une ressource protégée sans les privilèges adéquats.

Signification

La ressource demandée n'existe pas ou n'a pas été trouvée (ID Invalide).

Réponse

Recherche d'une ressource qui n'existe pas dans la base de données.

Signification

La méthode HTTP utilisée (GET, POST, PUT, DELETE) n'est pas autorisée pour cette ressource.

Réponse

Tentative d'utiliser une méthode DELETE sur une ressource où cette méthode n'est pas permise.

Signification

La requête est bien formée, mais elle ne peut pas être traitée en raison d'une erreur de validation.

Réponse

Tentative d'ajouter un utilisateur avec des données incomplètes ou non valides.

Signification

Une erreur serveur imprévue est survenue.

Réponse

Problème avec la base de données ou une erreur côté serveur.
